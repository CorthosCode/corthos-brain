# ☕ Многопоточность (глоссарий)

**\<u>Процессы (processes)\</u>** — изолированные, независимо выполняемые программы, которые пользуются ресурсами операционной системы, такими как память, дескрипторы файлов и учетные данные системы безопасности. При необходимости процессы могут взаимодействовать друг с другом с помощью коммуникационных механизмов: сокетов, обработчиков сигналов, совместной памяти, семафоров и файлов.



**\<u>Потоки выполнения (threads)\</u>** находятся внутри процессов, совместно используя общепроцессные ресурсы. Каждый поток выполнения имеет свой программный счетчик, стек и локальные переменные. Потоки обеспечивают естественную декомпозицию для задействования аппаратного параллелизма в многопроцессорных системах и внутри одной программы могут действовать одновременно в нескольких процессорах.



Потоки иногда называют **\<u>облегченными процессами (lightweight processes)\</u>**, и в большинствесовременных операционных систем именно они считаются основными единицами планирования. При отсутствии явной координации потоки выполняются одновременно и асинхронно по отношению друг к другу. Имея равный доступ к адресному пространству памяти процесса, все потоки используют одинаковые переменные и объекты из одной кучи, благодаря чему работают точнее межпроцессных механизмов. Но без явной синхронизации, обеспечивающей координацию доступа к совместным данным, потоки могут изменять переменные, которые уже используются другими потоками, с непредсказуемым результатом.



**\<u>Спецификация\</u>** в программировании — это формальный документ, который описывает язык программирования, программу или систему. Он определяет поведение, функции и требования к программе, обеспечивая точное понимание того, как она должна работать. Спецификации используются для стандартизации языков программирования и программных систем, чтобы разработчики и пользователи могли согласовывать ожидаемое поведение программ.



**\<u>Инварианты (invariants)\</u>** в программировании, особенно в объектно-ориентированном программировании (ООП), — это условия, которые должны оставаться истинными для объекта на протяжении всей его жизни. Они определяют допустимое внутреннее состояние объекта и гарантируют, что после вызова любых методов объекта его состояние остаётся согласованным и непротиворечивым. Инварианты часто не выражаются явно в коде, но могут быть проверены с помощью защитных конструкций.



**\<u>Постусловия (postcondition)\</u>** — это условия, которые должны быть выполнены после выполнения некоторой операции или метода. Они описывают эффекты от операций, то есть то, что должно быть истинным после их завершения. Постусловия используются для проверки того, что метод работает правильно и достигает ожидаемых результатов. В отличие от инвариантов, которые поддерживают состояние объекта между вызовами методов, постусловия фокусируются на результатах конкретных операций.



**\<u>Предусловие (precondition)\</u>****&#x20;**— это условие или набор условий, которые должны быть выполнены перед вызовом метода или функции, чтобы её выполнение было корректным. Предусловия определяют требования к состоянию объекта или входным данным на момент вызова метода. Если предусловия не выполняются, то метод может работать некорректно, выбрасывать исключения или даже приводить к неопределённому поведению программы.



**\<u>Спецификация\</u>** определяет **\<u>инварианты\</u>**, ограничивающие состояние объекта, и **\<u>постусловия\</u>**, описывающие **\<u>эффекты\</u>** от операций.



**\<u>Мемоизация\</u>****&#x20;**— это техника оптимизации, при которой результаты вычислений функции для определённых входных данных сохраняются (кешируются), чтобы избежать повторных вычислений для тех же входных данных в будущем. Если функция вызывается с уже обработанными ранее аргументами, результат берётся из кеша, а не вычисляется заново.



**\<u>Задача\</u>** (task) — абстрактная дискретная единица работы.



**\<u>Атомарность\</u>** — это свойство операции или действия быть неделимым, то есть выполняться как единое целое, без возможности прерывания или частичного выполнения.



**\<u>Взаимная блокировка (Deadlock)\</u>** — это ситуация, когда два или более потока навсегда ожидают друг друга, не отдавая удерживаемые ресурсы.



**\<u>Ресурсное голодание (Starvation)\</u>** — это ситуация, когда поток никогда не получает доступа к ресурсу, так как другие потоки постоянно его опережают.



**\<u>Ковариантность\</u>****&#x20;**— это свойство, при котором наследование типов сохраняется в производных типах в том же порядке. Например, если класс Cat наследуется от Animal, то коллекция Cat (например, IEnumerable\<Cat>) может рассматриваться как производный тип коллекции Animal (IEnumerable\<Animal>). Это позволяет использовать более конкретный (производный) тип там, где ожидается базовый.



**\<u>Инвариантность\</u>****&#x20;**— это ситуация, когда наследование исходных типов не переносится на производные типы. То есть, если есть два связанных класса, их обобщённые контейнеры не связаны отношением наследования, и для работы с ними требуется точное совпадение типов, например List\<Base>и List\<Derived>считаются разными типами и не совместимы.

```java
IEnumerable<Cat> cats = new List<Cat>();
// OK, ковариантность: можно использовать Cat вместо Animal, потому что IEnumerable только читает данные
IEnumerable<Animal> animals = cats;

List<Cat> catList = new List<Cat>();
// Ошибка! Инвариантность: списки для разных типов несовместимы, потому что в animalList можно положить не только котов, а любых животных
List<Animal> animalList = catList;
```
