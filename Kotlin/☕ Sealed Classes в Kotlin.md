## 1. Определение и цель

**Sealed class** (запечатанный класс) — это абстрактный класс, который ограничивает иерархию наследования. Все прямые подклассы запечатанного класса должны быть объявлены в том же файле (или модуле, начиная с Kotlin 1.5+), что и сам запечатанный класс.

**Главная цель:** Представление ограниченного набора типов (алгебраических типов данных, ADT). Это позволяет компилятору знать *все* возможные варианты реализации типа во время компиляции.

### Сравнение с Java
| Характеристика | Kotlin `sealed class` | Java `enum` | Java `sealed class` (Java 17+) |
| :--- | :--- | :--- | :--- |
| **Состояние** | Может иметь состояние (поля) | Фиксированные константы | Может иметь состояние |
| **Наследники** | Классы/Объекты/Data-классы | Только константы | Классы/Записи |
| **Гибкость** | Высокая (разные типы данных) | Низкая | Средняя/Высокая |
| **Когда появилось** | Kotlin 1.0 | Java 5 | Java 17 |

> **Для собеседования:** В Java до 17-й версии аналогом служили `enum` (если нет состояния) или паттерн "Маркерный интерфейс + проверка `instanceof`" (что небезопасно и многословно). С Java 17 появились нативные `sealed classes`, концепция идентична.

## 2. Синтаксис и правила

### Базовое объявление

```kotlin
sealed class Result {
    // Можно добавлять общие свойства или методы
    val timestamp: Long = System.currentTimeMillis()
}

// Подклассы могут быть:
// 1. Классами
class Success(val data: String) : Result()

// 2. Объектами (синглтоны)
object Error : Result()

// 3. Data-классами (наиболее часто)
data class Failure(val exception: Exception) : Result()
```

### Ключевые ограничения:

1.  **Конструктор запечатанного класса по умолчанию `private`**. Вы не можете создать экземпляр `Result()` напрямую.
2.  **Подклассы должны быть известны компилятору**.
    *   *До Kotlin 1.5:* Строго в одном файле.
    *   *Kotlin 1.5+:* В том же **модуле** (module). Это важно для больших проектов, но на собеседовании лучше уточнять версию языка. Обычно подразумевается ограничение видимости.
3.  **Нельзя наследовать от запечатанного класса вне его модуля/файла.**

## 3. Главное преимущество: Exhaustive `when`

Компилятор Kotlin проверяет полноту обработки вариантов в выражении `when`. Если вы обработали все подклассы, блок `else` не нужен. Если добавится новый подкласс — код **не скомпилируется**, пока вы не обновите `when`.

```kotlin
fun handleResult(result: Result): String {
    return when (result) {
        is Success -> "Data: ${result.data}"
        is Failure -> "Error: ${result.exception.message}"
        Error -> "Unknown error" 
        // else не нужен! Компилятор знает, что других вариантов нет.
    }
}
```

### Почему это важно для архитектуры?
Это реализует принцип **Pattern Matching** (сопоставления с образцом).
*   **Безопасность:** Невозможно забыть обработать новый тип ошибки.
*   **Рефакторинг:** При добавлении нового типа `Loading` в `sealed class Result`, IDE подсветит все места, где `when` стал неисчерпывающим.

> **Сравнение с Java:**
> В Java даже с `switch` по `enum` или `instanceof` checks компилятор не гарантирует обработку всех случаев без явного `default`. В Java 17+ с `sealed` и `switch` выражениями появляется похожая проверка (pattern matching switch), но в Kotlin это работает стабильнее и дольше.

## 4. Sealed Interfaces (Kotlin 1.5+)

Начиная с Kotlin 1.5, можно использовать `sealed interface`.

```kotlin
sealed interface UiState

data class Loading(val progress: Int) : UiState
data class Content(val data: List<String>) : UiState
object Empty : UiState
```

**Зачем нужно?**
Класс может реализовывать несколько интерфейсов. Если вам нужно, чтобы тип принадлежал к ограниченной иерархии, но при этом мог наследовать другой класс, используйте `sealed interface`.

```kotlin
class MyCustomView : View(), UiState { ... }
```
*(Примечание: `View` здесь условный пример базового класса).*

## 5. Продвинутые нюансы (для Senior уровня)

### А. Вложенные запечатанные классы
Можно вкладывать одни запечатанные классы в другие для создания древовидных структур состояний.

```kotlin
sealed class NetworkResult {
    sealed class Success : NetworkResult() {
        data class User(val name: String) : Success()
        data class Order(val id: Long) : Success()
    }
    object Error : NetworkResult()
}
```

### Б. Рефлексия и `sealedSubclasses`

У запечатанных классов есть свойство `.sealedSubclasses`, возвращающее список всех прямых подклассов.
```kotlin
Result::class.sealedSubclasses.forEach { println(it) }
```
*Использование:* Полезно для автоматической сериализации/десериализации (например, в Kotlin Serialization) или для построения динамических UI-компонентов.

### В. Производительность

`when` для запечатанных классов компилируется в эффективный байт-код.
*   Если подклассы — `object` (синглтоны), проверка идет через сравнение ссылок (`==`).
*   Если подклассы — классы, используется `instanceof` (или `invokevirtual` для проверки типа).
*   Это быстрее, чем цепочка `if-else instanceof` в Java, так как компилятор может оптимизировать таблицу переходов (tableswitch/lookupswitch), если вариантов много.

### Г. Совместимость с Java

Из Java запечатанный класс выглядит как обычный `abstract class` с приватным конструктором.
*   **Аннотация `@JvmOverloads`** тут не применима напрямую к иерархии, но важна для конструкторов подклассов, если они вызываются из Java.
*   В Java вы **не получите** преимуществ исчерпывающего `switch`. Вам придется использовать обычные проверки типов или библиотеки вроде Vavr.

## 6. Типичные кейсы использования (Best Practices)

1.  **Результаты операций (Result Pattern):**
    Вместо выбрасывания исключений или возврата `null`/`Optional`.
    ```kotlin
    sealed class ApiResult<out T> {
        data class Success<T>(val data: T) : ApiResult<T>()
        data class Error(val code: Int, val msg: String) : ApiResult<Nothing>()
    }
    ```

2.  **Состояния UI (State Management):**
    В Android (Jetpack Compose) или frontend-подобной логике.
    ```kotlin
    sealed class ScreenState {
        object Loading : ScreenState()
        data class ShowContent(val items: List<Item>) : ScreenState()
        data class ShowError(val message: String) : ScreenState()
    }
    ```

3.  **AST (Abstract Syntax Tree) или парсинг:**
    Когда нужно представить разные типы узлов дерева, которые обрабатываются рекурсивно.

4.  **Замена Enum с поведением:**
    Если `enum` становится слишком сложным (много методов, зависящих от константы), его заменяют на `sealed class` с объектами-наследниками.

## 7. Частые вопросы на собеседовании

**В: Чем отличается `sealed class` от `enum class`?**
**О:** `enum` — это фиксированный набор *констант* одного типа. У всех констант одинаковая структура данных. `sealed class` — это набор *типов*, каждый из которых может иметь свою уникальную структуру данных (поля, методы). `enum` нельзя параметризовать разными типами данных для разных констант гибко, а `sealed` можно.

**В: Можно ли создать экземпляр запечатанного класса вне файла объявления?**
**О:** Нет, конструктор приватный. Но можно создать экземпляры *подклассов*, если они `public` и доступны из текущего модуля (в Kotlin 1.5+).

**В: Что будет, если добавить новый подкласс и забыть обновить `when`?**
**О:** Ошибка компиляции (если `when` используется как выражение, т.е. возвращает значение). Если `when` используется как оператор (statement) и есть ветка `else`, ошибки не будет, но логика может стать неверной. *Совет:* Всегда старайтесь использовать `when` как выражение без `else` для sealed классов.

**В: Как сериализовать запечатанный класс в JSON?**
**О:** Стандартный `Gson`/`Jackson` не поймут полиморфизм "из коробки" без дополнительных аннотаций (`@JsonTypeInfo`). Kotlin Serialization поддерживает это нативно через полиморфную сериализацию, требуя регистрации подклассов.
